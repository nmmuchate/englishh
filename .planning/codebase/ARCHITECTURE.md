# Architecture

**Analysis Date:** 2026-02-20

## Pattern Overview

**Overall:** Quasar SPA with Vue 3 Composition API, organized as a layered client-side application with framework-provided boot sequence and router-based navigation.

**Key Characteristics:**
- Client-side Single Page Application using Vue 3 and Quasar framework
- Vite-based build system with SSR support capabilities
- Pinia store for state management with HMR support
- File-based routing with lazy-loaded layouts and pages
- Bootstrap sequence via boot files for application initialization

## Layers

**UI/Presentation Layer:**
- Purpose: Render user interface components and manage visual interactions
- Location: `src/pages/`, `src/components/`, `src/layouts/`
- Contains: Vue single-file components (`.vue` files) with templates, scripts, and scoped styles
- Depends on: Quasar UI components, state management (Pinia stores)
- Used by: Router, layouts, parent pages

**Routing Layer:**
- Purpose: Handle client-side navigation and URL-based view switching
- Location: `src/router/`
- Contains: Router configuration (`index.js`) and route definitions (`routes.js`)
- Depends on: Vue Router 5.x, layout and page components
- Used by: App.vue root component

**State Management Layer:**
- Purpose: Manage application state, getters, and actions in a centralized location
- Location: `src/stores/`
- Contains: Pinia store definitions (e.g., `example-store.js`)
- Depends on: Pinia framework with HMR support
- Used by: Components and pages that need shared state

**Bootstrap Layer:**
- Purpose: Initialize application dependencies and configure global properties on startup
- Location: `src/boot/`
- Contains: Boot files that execute before app mount (e.g., `axios.js`)
- Depends on: Framework wrappers, external libraries
- Used by: Application initialization sequence

**App Root:**
- Purpose: Mount point for Vue application, renders router view
- Location: `src/App.vue`
- Contains: Root component with `<router-view />` outlet
- Depends on: Router
- Used by: Quasar framework entry point

## Data Flow

**Application Initialization:**

1. Browser loads `index.html` with `<!-- quasar:entry-point -->` placeholder
2. Quasar build system injects main.js entry point
3. Boot files execute in sequence (e.g., `src/boot/axios.js`)
4. Pinia store is initialized via `src/stores/index.js`
5. Vue Router is initialized with routes from `src/router/routes.js`
6. App.vue is mounted as root component
7. Router renders matched layout/page components based on current URL

**Route Navigation:**

1. User navigates via URL or router link
2. Vue Router matches route to `routes.js` configuration
3. Router loads layout component (e.g., `MainLayout.vue`)
4. Router renders page component inside layout's `<router-view />`
5. Components access shared state via Pinia stores if needed

**API Communication:**

1. Boot file `src/boot/axios.js` creates axios instance with baseURL
2. Instance exported as `api` and available as `this.$api` in Options API
3. Components call API methods and update local/store state with responses

**State Management:**

- Stores use Pinia composition-style API with state, getters, and actions
- HMR (Hot Module Replacement) support via `import.meta.hot.accept()`
- Example store: `src/stores/example-store.js` (counter store with state, double-count getter, increment action)

## Key Abstractions

**Layout System:**
- Purpose: Provides shell UI structure (header, drawer, main content area)
- Examples: `src/layouts/MainLayout.vue`
- Pattern: Layout component wraps page content with `<router-view />`, defined in routes as parent of page routes

**Page Components:**
- Purpose: Represent full pages/routes in application
- Examples: `src/pages/IndexPage.vue`, `src/pages/ErrorNotFound.vue`
- Pattern: Defined as child routes in `routes.js`, lazy-loaded via dynamic import

**Reusable Components:**
- Purpose: Shared presentational components used across pages
- Examples: `src/components/EssentialLink.vue`
- Pattern: Props-based configuration, minimal state, imported directly in parent components

**Store (Pinia):**
- Purpose: Centralized state with getters and actions
- Examples: `src/stores/example-store.js`
- Pattern: `defineStore()` with state function, getters object, actions object; supports HMR

**Boot File:**
- Purpose: Execute initialization code before app mounts
- Examples: `src/boot/axios.js`
- Pattern: `defineBoot()` wrapper, receives `{ app }` context, sets global properties or initializes services

## Entry Points

**HTML Entry:**
- Location: `index.html`
- Triggers: Browser navigation to application root
- Responsibilities: Provides DOM mount point, loads favicon, defines meta tags

**JavaScript Entry:**
- Location: Auto-generated by Quasar (uses boot sequence → App.vue)
- Triggers: Build system creates main.js that imports App.vue, Router, and boots
- Responsibilities: Initialize Vue app, mount stores, execute boot files, start router

**Route Entry:**
- Location: `src/router/routes.js`
- Triggers: URL navigation matches route definition
- Responsibilities: Define which layout/page renders for given path

## Error Handling

**Strategy:** Catch-all route for unmatched paths

**Patterns:**
- 404 Error Page: `src/pages/ErrorNotFound.vue` matches `/:catchAll(.*)*` pattern
- Configured as last route in `routes.js` to catch all unmapped URLs
- Simple error page displayed when user navigates to non-existent route

## Cross-Cutting Concerns

**Logging:** Not explicitly configured; console methods available

**Validation:** Props validation in component definitions (e.g., `EssentialLink.vue` validates prop types)

**Authentication:** Not implemented in current codebase; infrastructure ready in Quasar for future auth integration

**Styling:**
- Global CSS: `src/css/app.css`
- Scoped styles: Within `<style scoped>` in individual `.vue` files
- Quasar CSS classes used for layout (e.g., `q-layout`, `q-page`, `flex flex-center`)

**API Integration:**
- Centralized axios instance in `src/boot/axios.js`
- BaseURL configurable via boot file
- Global access as `this.$api` in Vue components
- Export allows import in composition scripts: `import { api } from 'boot/axios'`
