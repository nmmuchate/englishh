```javascript

  { schedule: '0 2 * * *', timeZone: 'UTC', region: 'africa-south1' },
```

Implement `exports.deleteOldTranscripts`:
- Log start: `logger.info('deleteOldTranscripts: starting daily cleanup')`
- Compute cutoff: `const cutoff = admin.firestore.Timestamp.fromDate(new Date(Date.now() - 30 * 24 * 60 * 60 * 1000))`
- Query: `admin.firestore().collection('sessions').where('createdAt', '<', cutoff).get()` — this returns ALL matching docs; no `.select()` needed since we need refs for batch update
- Chunk into batches of 500 (Firestore limit): iterate `snap.docs` in slices of 500
- For each chunk: create a batch, call `batch.update(doc.ref, { transcript: admin.firestore.FieldValue.delete() })` for each doc, await `batch.commit()`
- Log: `logger.info(\`deleteOldTranscripts: cleaned \${snap.size} sessions\`)`

Implement `exports.updateWeeklyLeaderboard`:
- Schedule: `{ schedule: '0 0 * * 1', timeZone: 'UTC', region: 'africa-south1' }` (Monday 00:00 UTC)
- Log start: `logger.info('updateWeeklyLeaderboard: starting weekly reset')`
- Get current weekId using the existing `getWeekId()` helper already in functions/index.js
- Query top 100 users in current week: `admin.firestore().collection(\`leaderboard/${currentWeekId}/users\`).orderBy('weeklySessionTime', 'desc').limit(100).get()`
- If snapshot is empty, log and return early (nothing to archive)
- Calculate final ranks: map over `snap.docs` with index to assign `rank: index + 1`
- Archive: for each user doc, write to `leaderboard_archive/${currentWeekId}/users/${userId}` using a batch — include all fields from the user doc plus the computed rank. Use multiple batches if > 500 docs (in practice limited to 100 here)
- Create the archive week document: `admin.firestore().doc(\`leaderboard_archive/${currentWeekId}\`).set({ weekId: currentWeekId, archivedAt: admin.firestore.Timestamp.now(), totalParticipants: snap.size }, { merge: true })`
- Compute next weekId: call `getWeekId()` one week in the future using `new Date(Date.now() + 7 * 24 * 60 * 60 * 1000)` — create a local inline helper or compute directly
  - Actually: do NOT call getWeekId() — getWeekId() reads current date, so calling it after the current operations may still return the same week. Instead, get the new week doc by creating a fresh week document for the NEXT week. A simpler approach: create `leaderboard/{nextWeekId}` as a placeholder doc with `{ weekId: nextWeekId, createdAt: admin.firestore.Timestamp.now(), totalParticipants: 0 }`
  - Compute nextWeekId by duplicating getWeekId logic with `new Date(Date.now() + 7 * 24 * 60 * 60 * 1000)` inline
- Log: `logger.info(\`updateWeeklyLeaderboard: archived \${snap.size} users for \${currentWeekId}\`)`

Both functions must be wrapped in try/catch with logger.error on failure — onSchedule functions that throw will be retried by Cloud Scheduler.
